# unattended-consequences

Autonomous coding experiment. Claude researches a small business pain point,
designs a DDD-based solution, and builds it on Railway.

## Project Layout

- `specs/` — concept, architecture (domain model), aesthetic direction
- `src/` — application source code (organized by bounded context)
- `AGENTS.md` — how to build, run, test (operational only)
- `IMPLEMENTATION_PLAN.md` — prioritized task list (generated by planning mode)
- `GOAL.md` — mission brief and engineering philosophy
- `HUMAN_TASKS.md` — tasks requiring human action (account setup, API keys, etc.)
- `LOOP_LOG.md` — running self-assessment across build iterations

## Current Status

Not started. Run planning mode to research a problem and create specs.

## Rules

- Read IMPLEMENTATION_PLAN.md to decide what to work on
- Search before assuming something isn't implemented
- No placeholders or stubs — full implementations only
- Follow DDD: domain layer is pure, behavior on aggregates, explicit events
- DTU twins for external services, anti-corruption layers at boundaries
- Document human tasks in HUMAN_TASKS.md with clear instructions
- Write self-assessment to LOOP_LOG.md after each build iteration
- Update IMPLEMENTATION_PLAN.md with learnings after each task
- Update AGENTS.md with operational learnings (keep brief)
- For significant features: explore -> architect -> implement -> review (use feature-dev agents)
